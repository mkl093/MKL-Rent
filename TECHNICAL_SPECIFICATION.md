# Техническое задание

## Система учета оборудования для AV-рентала

**Статус:** согласованное ТЗ на MVP  
**Тип:** адаптивное веб-приложение  
**Среда разработки:** PyCharm + Claude Code  
**Архитектура:** модульный монолит  
**Основной стек:** Python, FastAPI, Uvicorn, PostgreSQL  
**Целевая организация:** одна рентальная компания  
**Язык интерфейса:** русский  
**Валюта:** EUR

---

## 1. Назначение

Разработать минималистичную систему для учета склада компании, сдающей в аренду аудио-, видео-, световое и другое AV/event-оборудование.

Основной рабочий процесс:

1. Ведение склада.
2. Создание проекта с датами аренды.
3. Проверка доступности оборудования.
4. Создание сметы.
5. Бронирование оборудования.
6. Создание packing-листа на основе сметы.
7. Комплектация вручную и сканированием штрих-кодов.
8. Автоматический расчет кейсов/рэков, веса и объема.
9. Формирование PDF-сметы и PDF packing-листа.

Система не должна пытаться быть полным аналогом Rentman. Главный принцип — минимальное количество экранов и действий при строгой складской логике.

## 4. Пользователи и вход

Система поддерживает несколько учетных записей. Все пользователи имеют одинаковые права, включая управление учетными записями.

Вход выполняется по уникальному логину и паролю.

Необходимо поддержать:

- вход и выход;
- создание пользователя;
- блокировку;
- удаление;
- сброс/изменение пароля.

Пароли хранятся только в виде безопасных хешей.

## 6. Структура склада

В MVP используется один общий склад без зон, стеллажей и полок. Архитектура должна позволять добавить места хранения позже.

### 6.1. Категории

Двухуровневая структура:

- категория;
- подкатегория.

Глубже двух уровней в MVP не требуется.

## 7. Модель оборудования

### 7.1. Обязательные поля

- категория;
- наименование модели;
- тип учета;
- вес одной единицы, кг;
- длина, ширина и высота одной единицы, мм;
- базовая цена аренды, EUR;
- признак наличия штатного кейса/рэка;
- тип упаковки, если она есть: кейс или рэк.

Для количественной модели также обязательно общее количество на складе.

Базовая цена может быть равна `0 EUR`.

### 7.2. Необязательные поля

- подкатегория;
- производитель;
- внутренний артикул;
- одно основное фото;
- описание;
- примечание.

### 7.3. Редактирование

После создания модель можно редактировать. Нельзя менять только тип учета.

Изменение модели не должно автоматически изменять уже созданные сметы и packing-листы. В документах должны храниться необходимые снимки данных на момент добавления.

### 7.4. Удаление и архив

- Неиспользованную модель можно удалить.
- Использованную модель можно только архивировать.
- Архивная модель недоступна для новых проектов.

---

## 8. Типы учета

### 8.1. Количественный

Хранится только общее количество без отдельных экземпляров.

- Штрих-коды не используются.
- В packing-лист добавляется вручную.
- Бронируется по количеству.

### 8.2. Посерийный

Каждая физическая единица представлена отдельным экземпляром.

Обязательные поля экземпляра:

- модель;
- уникальный штрих-код;
- статус.

Необязательные:

- серийный номер производителя;
- внутренний инвентарный номер;
- комментарий.

Вес, габариты и упаковка наследуются от модели.

---

## 9. Статусы оборудования

- **Активно**
- **В ремонте**
- **Списано**

Статус «Забронировано» рассчитывается автоматически по проектам.

Для экземпляра хранится история статусов:

- дата и время;
- пользователь;
- старый статус;
- новый статус;
- комментарий.

Экземпляр можно удалить только если он нигде не использовался. В противном случае используется статус «Списано».

---

## 10. Количественные остатки

Количество можно быстро изменить в карточке модели. Каждое изменение сохраняется:

- дата и время;
- пользователь;
- старое количество;
- новое количество;
- разница;
- необязательный комментарий.

Отдельный модуль приходов/списаний в MVP не требуется.

---

## 11. Фото

Для модели допускается одно фото.

Поддерживаются JPG, PNG, WebP. Максимальный размер исходного файла — 10 МБ. После загрузки изображение оптимизируется с сохранением пропорций.

Фото показывается:

- в карточке модели;
- в списке склада;
- в списке выбора для сметы.

В PDF и packing-листе фото не выводится.

---

## 12. Штатный кейс или рэк

У модели может быть только одно правило упаковки.

Если упаковка есть, обязательны:

- тип: кейс или рэк;
- вес пустой упаковки, кг;
- длина, ширина и высота, мм;
- вместимость в единицах оборудования.

Название формируется автоматически:

- `Кейс для [модель]`;
- `Рэк для [модель]`.

Кейсы/рэки в MVP:

- не имеют складского остатка;
- не бронируются;
- не сканируются;
- рассчитываются автоматически;
- участвуют в весе и объеме;
- отображаются в packing-листе;
- не отображаются как оплачиваемая позиция сметы.

Количество упаковок:

```text
ceil(количество в упаковке / вместимость)
```

Распределение по отдельным кейсам не показывается. Достаточно общей строки, например `2 кейса`.

---

## 13. Проект

Проект — центральная сущность. Смета, бронирование, packing-лист и PDF принадлежат проекту.

### 13.1. Обязательные поля

- название;
- дата начала аренды;
- дата окончания аренды;
- общий коэффициент аренды;
- VAT.

### 13.2. Необязательные поля

- заказчик текстом;
- адрес;
- комментарий.

Отдельного справочника клиентов нет.

### 13.3. Даты

Используются только календарные даты без времени. Обе даты включаются в период бронирования.

Если один проект заканчивается 10 июля, а другой начинается 10 июля, это пересечение.

### 13.4. Статусы

- **Черновик**
- **Забронирован**
- **Завершен**
- **Отменен**

Логика:

- Черновик не резервирует оборудование.
- Забронирован резервирует оборудование.
- Завершен освобождает бронь и уходит в архив.
- Отменен не резервирует оборудование и уходит в архив.

Дата окончания сама по себе бронь не освобождает. Проект можно завершить в любой момент.

### 13.5. Изменение дат

При изменении дат забронированного проекта:

1. доступность пересчитывается;
2. требуется повторное подтверждение;
3. при дефиците требуется отдельное подтверждение.

### 13.6. Архив

Завершенные и отмененные проекты автоматически скрываются в архиве.

Архивный проект доступен только для просмотра. Для редактирования его нужно вернуть в статус «Черновик».

### 13.7. Удаление

Проект можно удалить только если:

- он в статусе «Черновик»;
- packing-лист еще не создан.

### 13.8. Копирование

Копируются общие данные и смета. Не копируются даты, packing-лист и статус бронирования. Новый проект создается как «Черновик».

## 14. Нумерация

Автоматические номера:

- проект: `PRJ-YYYY-NNN`;
- смета: `EST-YYYY-NNN`;
- packing-лист: `PL-YYYY-NNN`.

Счетчики раздельные, уникальные и сбрасываются каждый календарный год. Ручное изменение номера запрещено.

## 15. Доступность и бронирование

Для модели на выбранный период отображаются:

- всего;
- недоступно по статусу;
- забронировано в других проектах;
- доступно;
- требуется текущему проекту;
- дефицит.

Пользователь может открыть детализацию занятости:

- номер и название проекта;
- даты;
- статус;
- забронированное количество;
- ссылка на проект.

Для посерийной модели при бронировании резервируется количество по модели. Конкретные экземпляры назначаются позже в packing-листе.

Недоступное оборудование не скрывается. Его можно добавить в смету. При переводе проекта в «Забронирован» дефицит требует отдельного подтверждения и записывается в журнал.

## 16. Смета

### 16.1. Добавление оборудования

Внутри сметы пользователь:

1. выбирает категорию;
2. выбирает подкатегорию;
3. получает список моделей;
4. указывает количество;
5. добавляет выбранные позиции.

В списке видны:

- фото;
- модель;
- производитель;
- доступность на даты проекта;
- базовая цена;
- тип учета;
- наличие кейса/рэка.

Должен быть поиск по названию и производителю.

### 16.2. Массовое добавление

Можно выбрать несколько моделей, указать количество для каждой и нажать **«Добавить выбранное»**.

Если модель уже есть в смете, система предлагает:

- увеличить количество в существующей строке;
- создать отдельную строку.

### 16.3. Поля строки

- модель;
- количество;
- цена за единицу;
- коэффициент;
- скидка строки, %;
- итог;
- комментарий;
- порядок сортировки.

Базовая цена подставляется из модели, но редактируется внутри проекта.

### 16.4. Коэффициент аренды

В проекте вручную задается общий коэффициент. Для отдельной строки его можно переопределить.

```text
итог строки = цена единицы × количество × коэффициент строки × (1 − скидка строки / 100)
```

Автоматической таблицы коэффициентов по дням нет.

### 16.5. Произвольные строки

Можно добавить строку, не связанную со складом:

- наименование;
- количество;
- цена;
- коэффициент;
- комментарий.

Она участвует только в стоимости и PDF-смете. Не участвует в бронировании, packing-листе, весе и объеме.

### 16.6. Группировка

Смета группируется по категориям. Внутри категории порядок строк можно менять вручную. Этот порядок используется в PDF.

### 16.7. Скидка, VAT и валюта

- Валюта фиксирована: EUR.
- Поддерживается индивидуальная скидка по каждой строке (в %). Она применяется
  к строке до общей скидки и уже учтена в итоге строки (§16.4).
- Поддерживается общая скидка сметы. Она применяется к подытогу (сумме итогов строк).
- При скидке `0%` строка скидки (строчная или общая) не отображается.
- VAT по умолчанию задается в настройках.
- VAT можно изменить в конкретном проекте.

### 16.8. Версии

В MVP хранится только актуальное состояние сметы. Версионирование будет добавлено позже.

## 17. Packing-лист

### 17.1. Создание

Packing-лист создается вручную кнопкой **«Создать packing-лист»** на основе текущей сметы.

При создании:

- копируются только складские позиции;
- одинаковые модели из нескольких строк сметы объединяются;
- количества суммируются;
- произвольные строки сметы не копируются;
- цены и коэффициенты не переносятся.

### 17.2. Независимость от сметы

После создания packing-лист существует отдельно.

Изменение или удаление строки в смете:

- не меняет packing-лист автоматически;
- показывает расхождение;
- предлагает синхронизацию;
- применяет ее только после подтверждения.

### 17.3. Порядок

Packing-лист группируется по категориям и подкатегориям. Порядок внутри группы можно менять вручную. Он используется в PDF.

### 17.4. Статусы

- **Не начат**
- **Комплектуется**
- **Скомплектован**

При полном соответствии статус можно установить обычно.

При недокомплекте:

- требуется отдельное подтверждение;
- обязателен комментарий с причиной.

Packing-лист можно без ограничений вернуть из «Скомплектован» в «Комплектуется».

### 17.5. План и факт

Для строки показываются:

- плановое количество;
- фактическое количество;
- недостающее количество;
- количество сверх плана;
- статус.

### 17.6. Количественное оборудование

Добавляется только вручную: модель + количество.

Складское оборудование добавляется в packing-лист через подбор **«Добавить оборудование»**
(так же, как в смете): поиск и фильтр по категории, доступность на даты проекта, выбор
количества по каждой модели. Если модель уже есть в листе, увеличивается плановое
количество — отдельные строки не задваиваются (иначе сломалось бы сканирование серийных).
Серийные модели добавляются так же — создаётся строка, а экземпляры назначаются по §17.7.
Добавление сверх плана сметы допустимо; смета при этом не меняется (§17.8).

### 17.7. Серийное оборудование

Конкретный экземпляр добавляется:

- сканированием;
- ручным выбором из списка.

В списке видны штрих-код, серийный номер при наличии и статус.

### 17.8. Сверх плана

При превышении плана система предупреждает и требует подтверждение. Позиция помечается как «Сверх плана». Смета не меняется.

### 17.9. Дополнительные позиции

Можно добавить свободную строку:

- наименование;
- количество;
- вес;
- размеры или объем;
- комментарий.

Она не участвует в бронировании и смете, но может участвовать в общем весе и объеме.

---

## 18. Распределение по упаковке

Для каждой строки:

- количество в упаковке;
- количество без упаковки.

Их сумма должна быть равна общему количеству строки.

Если у модели есть кейс/рэк, при создании packing-листа все количество по умолчанию считается упакованным. Пользователь может перенести часть в «Без упаковки».

Количество упаковок считается только для упакованной части.

---

## 19. Вес

Единица — кг. Отображение — один знак после запятой.

Для строки показываются:

- вес оборудования;
- вес упаковки;
- общий вес.

```text
вес оборудования = количество × вес единицы

вес упаковки =
ceil(количество в упаковке / вместимость)
× вес пустой упаковки

общий вес = вес оборудования + вес упаковки
```

Внизу packing-листа показывается общий вес проекта.

---

## 20. Объем

Размеры хранятся в мм, объем отображается в м³.

```text
объем единицы, м³ =
(длина_мм × ширина_мм × высота_мм) / 1_000_000_000
```

Для строки показываются:

- объем оборудования без упаковки;
- объем кейсов/рэков;
- общий объем.

Упакованное оборудование учитывается только через внешний объем упаковки. Внизу показывается общий объем проекта.

---

## 21. Штрих-коды

### 21.1. Правила

В MVP штрих-коды не генерируются. Они вводятся вручную при создании серийного экземпляра.

Штрих-код:

- обязателен для серийного экземпляра;
- уникален во всей системе;
- проверяется на дублирование.

### 21.2. Форматы

- Code 128;
- Code 39;
- EAN-13;
- EAN-8;
- QR.

Основной формат — Code 128.

### 21.3. Ввод

- камера смартфона/планшета;
- ручной ввод;
- USB/Bluetooth-сканер как клавиатура.

### 21.4. Глобальный поиск

В разделе склада доступен поиск по штрих-коду. После ввода/сканирования открывается карточка экземпляра с моделью, статусом, серийным номером, штрих-кодом и историей.

## 22. Мобильное сканирование

После успешного сканирования экземпляр сразу добавляется в packing-лист.

Система показывает:

- крупное визуальное подтверждение;
- модель;
- штрих-код;
- прогресс комплектации;
- звуковой сигнал успеха.

При ошибке используются другое уведомление и другой звук. Звук можно отключить.

Необходимо блокировать:

- повторное добавление;
- списанный экземпляр;
- экземпляр в ремонте;
- конкурентное добавление двумя пользователями.

Проверка выполняется на сервере в транзакции.

Доступна команда **«Отменить последнее сканирование»**.

При потере связи:

- сканирование не считается успешным;
- показывается предупреждение;
- последнее действие можно повторить вручную после восстановления связи;
- полноценной офлайн-очереди нет.

18. Упаковка участвует в весе и объеме.
19. Все пользователи имеют одинаковые права.
20. Сервер предотвращает конфликты при совместной работе.

## 45. Правила для Claude Code

### 45.1. Перед изменениями

Claude Code обязан прочитать:

- `TECHNICAL_SPECIFICATION.md`;
- `CLAUDE.md`;
- `README.md`;
- релевантные модели;
- миграции;
- тесты.

### 45.2. Не расширять scope

Без прямого указания не добавлять:

- счета;
- клиентов как сущность;
- выдачу/возврат;
- роли;
- мультисклад;
- React;
- Redis;
- Celery;
- микросервисы;
- календарь;
- импорт/экспорт.

### 45.3. Порядок работы

Для каждой задачи:

1. изучить существующий код;
2. перечислить затрагиваемые файлы;
3. дать краткий план;
4. внести минимальные изменения;
5. создать миграцию при изменении БД;
6. добавить тесты;
7. запустить тесты;
8. описать результат и ограничения;
9. обновить документацию.

### 45.4. Требования к коду

- type hints;
- небольшие функции;
- бизнес-логика в services/domain layer;
- `Decimal` для денег;
- не использовать `float` для денежных сумм;
- транзакции для критических операций;
- timestamp хранить в UTC, отображать в выбранном timezone;
- не редактировать примененные миграции задним числом;
- не удалять тесты ради прохождения сборки;
- не дублировать формулы в разных слоях.

### 45.5. Definition of Done

Задача завершена, когда:

- функционал работает;
- миграции применяются на чистой БД;
- тесты проходят;
- линтинг/форматирование проходят;
- нет секретов и локальных путей;
- интерфейс проверен на десктопной и мобильной ширине;
- ошибки понятны пользователю;
- документация обновлена.

```markdown
1. Read TECHNICAL_SPECIFICATION.md before implementation.
2. This is a modular monolith built with FastAPI, PostgreSQL, SQLAlchemy 2,
   Alembic, Jinja2, HTMX and Bootstrap 5.
3. Do not introduce React, Flask, Redis, Celery or microservices.
4. Keep business logic out of route handlers and templates.
5. Use Decimal for all money calculations.
6. Add tests for every business rule.
7. Use PostgreSQL constraints and transactions for barcode uniqueness,
   reservations and packing scans.
8. Do not change the approved MVP scope without explicit instruction.
9. Before coding, inspect existing models, services, migrations and tests.
10. After coding, run tests and summarize changed files.
```

---

## 47. PyCharm

Рекомендуется:

- открыть корень Git-репозитория как проект;
- использовать Docker Compose interpreter или локальный virtualenv;
- создать Run Configurations для Uvicorn, pytest, Alembic upgrade/revision и backup;
- включить inspections и type checking;
- не добавлять `.env` в Git;
- использовать одинаковые команды форматирования и тестов в PyCharm и CLI.

---

## 48. Будущее развитие

Позже можно добавить:

- места хранения;
- учет кейсов/рэков как отдельных единиц;
- выдачу и возврат;
- импорт/экспорт;
- версии смет;
- календарь загрузки;
- роли;
- несколько складов;
- внешний API;
- интеграции;
- офлайн-очередь;
- realtime-обновления.

Эти функции не должны реализовываться заранее.
